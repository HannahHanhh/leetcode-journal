# Claude 项目规则 — leetcode-journal

这是一个刷题日志 + SM-2 复习调度仓库，**不是代码仓库**。用户在 LeetCode 网站上做题和看题解，这里只维护学习记录和复习节奏。

## 用户偏好
- 语言：Python（仅限脚本；不为题目写代码模板）
- 目标：LeetCode 热题 100
- 协作风格：**规划优先**——帮他排题、调度、复盘，不手把手讲每道题
- 遇到具体题目时默认**提示模式**（Hint，不直接给答案），除非他要求讲解

## 核心工作流
1. 用户来时，先跑 `python scripts/review.py today`，告诉他今天要复习什么。
2. 建议下一道新题时，按 `plan/hot100.md` 顺序推荐，同专题连做。
3. 用户报告做完某题 → 帮他：
   - （新题）`review.py add <id> "<title>"` + `review.py log <id> <grade>`
   - （旧题）`review.py log <id> <grade>`
   - 询问他是否要写题记（`problems/<id>-<slug>.md`）
   - 写 Git commit

## SM-2 评分（0-5）
引导用户给出诚实的评分。用户常见的自然语言 → 评分映射：
- "秒出 / 一眼" → 5
- "想了一下 / 有点犹豫" → 4
- "费劲但做出来了 / 卡了比较久" → 3
- "错了但看答案秒懂" → 2
- "错了，只记得一点思路" → 1
- "完全不会" → 0

**q<3 会重置 reps（间隔回到 1 天）**——提醒用户这不是惩罚，是记忆科学。

## 题记（problems/*.md）写什么
- 题目链接
- **用户自己的一句话思路**（不是抄题解）
- 数据结构 / 算法模式
- 易错点、踩过的坑
- 时间/空间复杂度
- 刷题历史（由 commit 和 schedule.json 记录，不用手动写）

不要写代码——用户在 LeetCode 网站上做。

## Commit 规范
- `solve: 0001 two-sum (hash)` — 新题首刷
- `review: 0001 grade=5` — 每次复习
- `note: 0001 <简述补充>` — 题记更新
- `plan: <动作>` — 改计划文档
- `chore: <动作>` — 改脚本/配置

## 不要做的事
- 不要为题目写 Python 代码实现或测试模板
- 不要直接给解法（除非用户明确要求讲解）
- 不要改 `review/schedule.json`（只通过 `review.py` 改）
- 不要批量加题——一道一道加，保证复习节奏真实反映做题节奏
